# 开发环境搭建

> 面向从没配过嵌入式环境的同学。本文所有步骤都可以照抄，命令直接复制到终端执行即可。
> 目标芯片：**STM32F103C8T6**。工具链：**CMake + arm-none-eabi-gcc + Ninja**，用 **VSCode** 开发，**ST-Link + OpenOCD** 烧录调试。

---

## 0. 总览

在开始之前，你需要在电脑上装好这几样东西（后面每个系统都有详细步骤）：

| 工具 | 用途 | 版本要求 |
| --- | --- | --- |
| **Git** | 拉代码、获取 HAL | 任意较新版本 |
| **CMake** | 构建系统 | **≥ 3.21**（用到了 CMakePresets） |
| **Ninja** | 构建执行器（比 make 快） | 任意较新版本 |
| **arm-none-eabi-gcc** | ARM 交叉编译器（编固件） | GNU Arm Embedded Toolchain |
| **OpenOCD** | 烧录 / 调试 STM32 | ≥ 0.11 |
| **VSCode** | 代码编辑器（推荐） | 最新版 |
| host 编译器（gcc/clang/MSVC） | 在 PC 上编译 algorithms 并跑单元测试 | 可选，但强烈建议 |

其中 **HAL 库**（STM32CubeF1）不入库，由 `scripts/fetch_hal.sh` **按需获取**（只 sparse-clone `Drivers` 目录，约数十 MB）到 `core/STM32CubeF1/`；`bootstrap` 脚本会自动帮你完成，只想跑 host 测试则完全不需要它。

### 最快路径（装好上面工具后）

```bash
git clone <仓库地址> health-terminal
cd health-terminal
./scripts/bootstrap.sh      # Linux / macOS
# 或 Windows PowerShell：
# .\scripts\bootstrap.ps1
```

`bootstrap` 脚本会自动检查工具是否齐全、获取 HAL、并完成首次配置。跑通它基本就代表环境 OK。

## 1. Windows 安装（重点，多数同学用 Windows）

Windows 下逐个安装。装完每一步都建议按提示做**验证命令**，看到版本号才算成功。

> 提示：安装涉及 PATH 的工具后，**要新开一个终端**（关掉旧的重开），环境变量才会生效。

### 1.1 Git for Windows

到 <https://git-scm.com/download/win> 下载安装包，一路默认下一步即可。验证：

```bash
git --version
```

### 1.2 CMake

到 <https://cmake.org/download/> 下载 Windows x64 Installer（`.msi`）。安装时**务必勾选 "Add CMake to the system PATH for all users"**。验证：

```bash
cmake --version
```

要求显示 **3.21 或更高**。

### 1.3 Ninja

三选一：

- **用 scoop**：`scoop install ninja`
- **用 choco**：`choco install ninja`
- **手动下载**：到 <https://github.com/ninja-build/ninja/releases> 下载 `ninja-win.zip`，解压出 `ninja.exe`，放到一个固定目录（例如 `C:\tools\ninja\`），再把该目录加入系统 PATH。

验证：

```bash
ninja --version
```

### 1.4 Arm GNU Toolchain (arm-none-eabi)

到 ARM 官网 <https://developer.arm.com/downloads/-/arm-gnu-toolchain-downloads> 下载 **arm-none-eabi** 的 Windows `.exe` 安装包。安装最后一步**务必勾选 "Add path to environment variable"**。验证：

```bash
arm-none-eabi-gcc --version
```

看到类似 `arm-none-eabi-gcc (Arm GNU Toolchain ...) 13.x` 即成功。

### 1.5 OpenOCD

Windows 没有官方安装器，二选一：

- **xpack-openocd（推荐）**：<https://github.com/xpack-dev-tools/openocd-xpack/releases>，下载 win32-x64 的压缩包。
- **openocd-mingw**：<https://gnutoolchains.com/arm-eabi/openocd/>。

解压后把里面的 `bin` 目录加入系统 PATH。验证：

```bash
openocd --version
```

### 1.6 host 侧编译器（跑单元测试用）

为了能在 PC 上编译 `algorithms` 并跑单元测试（**不需要开发板**），需要一个本机 C/C++ 编译器，二选一：

- **MSVC Build Tools（推荐）**：下载 "Build Tools for Visual Studio" <https://visualstudio.microsoft.com/downloads/>，安装时勾选 "使用 C++ 的桌面开发"。
- **MinGW-w64 gcc**：`scoop install gcc` 或 `choco install mingw`。

### 1.7 ST-Link 驱动

插上 ST-Link 后 Windows 需要正确驱动，二选一：

- **ST 官方驱动**：搜索 "STSW-LINK009"，下载安装。
- **Zadig**：<https://zadig.akeo.ie/>，选中 ST-Link 设备，安装 WinUSB 驱动（配合 OpenOCD 更省心）。

### 1.8 替代方案：用 scoop 一键装齐

如果嫌上面一个个装麻烦，装好 [scoop](https://scoop.sh/) 后可以一条命令搞定大部分：

```bash
scoop bucket add main
scoop install git cmake ninja gcc-arm-none-eabi openocd
```

（host 编译器 MSVC / MinGW 和 ST-Link 驱动仍需按上面单独装。）

## 2. macOS 安装

用 [Homebrew](https://brew.sh/)：

```bash
brew install cmake ninja open-ocd
brew install --cask gcc-arm-embedded
```

如果 `--cask gcc-arm-embedded` 装不上，可改用：

```bash
brew install arm-none-eabi-gcc
```

host 侧编译器直接用系统自带的 **clang**（装好 Xcode Command Line Tools 即可）：

```bash
xcode-select --install
```

验证：

```bash
cmake --version
ninja --version
arm-none-eabi-gcc --version
openocd --version
clang --version
```

## 3. Linux (Ubuntu / Debian) 安装

一条命令装齐：

```bash
sudo apt update
sudo apt install git cmake ninja-build gcc-arm-none-eabi gdb-multiarch openocd build-essential
```

验证：

```bash
cmake --version        # 需 ≥ 3.21，若发行版自带过旧，见下方提示
ninja --version
arm-none-eabi-gcc --version
openocd --version
```

> 如果 apt 里的 CMake 版本低于 3.21，可用 pip 装新版：`pip install --user cmake`，或使用 Kitware 官方 APT 源。

## 4. VSCode 配置

### 4.1 安装扩展

在 VSCode 扩展市场搜索并安装：

- **C/C++**（`ms-vscode.cpptools`）
- **CMake Tools**（`ms-vscode.cmake-tools`）
- **Cortex-Debug**（`marus25.cortex-debug`）— 用于烧录调试

### 4.2 选择 Preset 构建

用 CMake Tools 时，底部状态栏或命令面板（`Ctrl+Shift+P` → `CMake: Select Configure Preset`）里可以直接选择本仓库的 preset：

- `host` — 在 PC 上编译并跑单元测试
- `stm32` — ARM 交叉编译生成固件

选好 preset 后点状态栏的 **Build** 即可编译。

### 4.3 Cortex-Debug 调试配置

在项目根目录创建（或补充）`.vscode/launch.json`，写入如下 Cortex-Debug 配置：

```json
{
  "version": "0.2.0",
  "configurations": [
    {
      "name": "Debug STM32 (OpenOCD)",
      "type": "cortex-debug",
      "request": "launch",
      "servertype": "openocd",
      "cwd": "${workspaceFolder}",
      "executable": "build/stm32/firmware.elf",
      "device": "STM32F103C8",
      "interface": "swd",
      "configFiles": [
        "interface/stlink.cfg",
        "target/stm32f1x.cfg"
      ],
      "runToEntryPoint": "main",
      "svdFile": "STM32F103.svd"
    }
  ]
}
```

> `svdFile` 是可选的（用于寄存器视图）；没有对应 svd 文件时可以把这行删掉。
> 先确保已经执行过第 7 节的交叉编译，`build/stm32/firmware.elf` 存在，才能调试。

## 5. 拉取代码与获取 HAL

先正常克隆仓库：

```bash
git clone <仓库地址> health-terminal
cd health-terminal
```

HAL 库（STM32CubeF1）**不入库**，编译固件前需获取一次（只想跑 host 测试可跳过）。表现为 `core/STM32CubeF1` 不存在时，执行：

```bash
bash scripts/fetch_hal.sh
```

> 该脚本用 `--depth 1 --filter=blob:none --sparse` 只拉 CubeF1 的 `Drivers` 目录（HAL + CMSIS），
> 约数十 MB，比整仓（含大量例程，>1GB）快得多。若网络较慢请耐心等待。

也可以直接运行一键脚本，它内部会自动获取 HAL：

```bash
./scripts/bootstrap.sh      # Linux / macOS
.\scripts\bootstrap.ps1     # Windows PowerShell
```

## 6. 编译 host 单元测试（人人都能跑，不需要开发板）

这一步在你自己的电脑上编译并运行 `algorithms` 的单元测试，**不需要任何硬件**，是验证环境是否配好的最简单方式。

```bash
cmake --preset host
cmake --build --preset host
ctest --preset host --output-on-failure
```

预期结果：`test_filters`、`test_codec`、`test_algorithms` 全部显示 **Passed**，类似：

```text
100% tests passed, 0 tests failed out of 3
```

（构建产物在 `build/host/` 目录下。）

## 7. 交叉编译固件

这一步生成能烧进 STM32 的固件：

```bash
cmake --preset stm32
cmake --build --preset stm32
```

产物在 `build/stm32/` 目录：

- `firmware.elf` — 带调试信息，供 OpenOCD/GDB 调试用
- `firmware.bin` — 纯二进制
- `firmware.hex` — Intel HEX 格式

编译结束会打印 Flash / RAM 占用，类似：

```text
   text    data     bss     dec     hex filename
  12345     120    2048   14513    38b1 firmware.elf
```

### MODULE_SET：只编自己负责的模块

`stm32` preset 有一个缓存变量 `MODULE_SET`，用来选择要编译的模块子集，避免每个人都编全部代码。可选值：`yuhao` | `chafanting` | `liuyanming` | `full`。例如只编译 yuhao 的模块：

```bash
cmake --preset stm32 -DMODULE_SET=yuhao
cmake --build --preset stm32
```

`full` 表示编译全部模块（集成/联调时用）。不指定时使用 preset 里的默认值。

## 8. 烧录与调试（ST-Link + OpenOCD）

### 8.1 接线

**ST-Link ↔ 开发板（SWD 4 线）：**

| ST-Link | STM32F103 |
| --- | --- |
| SWDIO | PA13 |
| SWCLK | PA14 |
| GND | GND |
| 3.3V | 3V3 |

**USB-TTL 串口 ↔ 开发板（看日志用）：**

| USB-TTL | STM32F103 |
| --- | --- |
| RX | PA9 (USART1 TX) |
| GND | GND |

> 注意 **TX 接 RX 是交叉的**：开发板的 TX(PA9) 接串口模块的 RX。

### 8.2 一条命令烧录

```bash
openocd -f interface/stlink.cfg -f target/stm32f1x.cfg -c "program build/stm32/firmware.elf verify reset exit"
```

看到 `Verified OK` 和 `** Resetting Target **` 就烧录成功了。

### 8.3 起 GDB Server 调试

先在一个终端启动 OpenOCD（它会监听 3333 端口的 gdb server）：

```bash
openocd -f interface/stlink.cfg -f target/stm32f1x.cfg
```

再开另一个终端连上去调试：

```bash
arm-none-eabi-gdb build/stm32/firmware.elf
# 进入 gdb 后：
(gdb) target extended-remote localhost:3333
(gdb) load
(gdb) monitor reset halt
(gdb) continue
```

> 用 VSCode 的话，直接按第 4.3 节配好的 Cortex-Debug 按 F5 即可，不用手敲 gdb。

### 8.4 首版固件烧录后的现象

固件正常运行时：

- **PC13 上的 LED 会闪烁**（板载 LED，用来判断程序有没有跑起来）。
- **USART1 会打印版本号**，串口参数为 **115200 波特率、8 数据位、无校验、1 停止位（8N1）**。

用串口助手 / `putty` / `screen` 查看输出，例如 Linux/macOS：

```bash
screen /dev/ttyUSB0 115200
```

（退出 screen：`Ctrl+A` 然后按 `K`，再按 `Y`。）Windows 上直接用 PuTTY 选对应 COM 口、波特率 115200 即可。

## 9. 常见问题 FAQ

**Q1. `arm-none-eabi-gcc: command not found` / 不是内部或外部命令**
说明编译器没进 PATH。重新检查安装时是否勾选了 "Add path to environment variable"，或手动把 toolchain 的 `bin` 目录加入 PATH，然后**新开终端**再试。验证：`arm-none-eabi-gcc --version`。

**Q2. `ninja: command not found` / CMake 报 "Could not find Ninja"**
Ninja 没装或没进 PATH。按 1.3 节重新安装，把 `ninja.exe` 所在目录加入 PATH，新开终端后 `ninja --version` 能出版本号即可。

**Q3. 配置 stm32 时报错找不到 HAL / `firmware.cmake` 找不到 STM32CubeF1**
基本都是 **HAL 尚未获取**（`core/STM32CubeF1` 不存在或为空）。执行：

```bash
bash scripts/fetch_hal.sh
```

拉完再重新 `cmake --preset stm32`。

**Q4. OpenOCD 找不到 ST-Link（`Error: open failed` / `unable to find a matching CMSIS-DAP device`）**
- Windows：驱动没装对，用 ST 官方驱动或 Zadig 装 WinUSB（见 1.7）。
- Linux：多为**权限问题**，需要加 udev 规则。创建 `/etc/udev/rules.d/49-stlink.rules`，写入：

  ```text
  SUBSYSTEM=="usb", ATTR{idVendor}=="0483", ATTR{idProduct}=="3748", MODE="0666"
  SUBSYSTEM=="usb", ATTR{idVendor}=="0483", ATTR{idProduct}=="374b", MODE="0666"
  ```

  然后 `sudo udevadm control --reload-rules && sudo udevadm trigger`，重新插拔 ST-Link。

**Q5. 串口没有任何输出**
逐项排查：① 波特率是否设成 **115200 8N1**；② **TX/RX 是否接反**（开发板 PA9=TX 要接串口模块的 RX）；③ GND 是否共地；④ 固件是否真的烧进去了（PC13 LED 有没有闪）；⑤ 选对了串口设备（Windows 看设备管理器的 COM 号，Linux 一般是 `/dev/ttyUSB0`）。

**Q6.（补充）CMake 版本过低报错，不认识 `CMakePresets.json`**
本项目要求 CMake ≥ 3.21。用 `cmake --version` 确认，过低就升级（Windows 重装新版 installer；Linux 用 `pip install --user cmake`）。
