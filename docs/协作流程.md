# Git 协作流程

> 团队：王宇浩（组长）、查樊听、刘晏铭
> 背景：7~8 月异地分散开发（各自开发板 + 各自传感器），9 月初集中一周做整机集成。
>
> 里程碑：`7.8 开题` → `7 月分散开发` → `8.10 中期检查（交 PPT + 各模块实测数据）` → `9 月初集中集成` → `9.8 最终验收`

本流程要解决的核心问题：在各人改各人模块的前提下尽量减少冲突，保证 dev/main 始终可编译可演示，
并让每个里程碑都有清晰、可回溯的版本。

---

## 1. 分支模型

采用三层分支，各司其职：

| 分支 | 定位 | 规则 |
| --- | --- | --- |
| `main` | 受保护的稳定分支 | 仅存放**可编译、可演示**的版本；只能从 `dev` 合入；里程碑在此打 tag。禁止直接 push。 |
| `dev` | 集成主线 | 各人 feature 合入于此；每周合并一次，合并后 host 测试须全绿。 |
| `feature/姓名-模块` | 个人功能分支 | 从 `dev` 切出，完成后合回 `dev`。 |

个人分支命名示例：`feature/yuhao-max30102`、`feature/chafanting-hx711-weight`、`feature/liuyanming-oled`。

分支关系示意：

```
                       tag: v0.5-midterm            tag: v1.0-final
                              │                            │
main    ●─────────────────────●────────────────────────────●─────────►
         \                   /   \                        /
          \   (每周稳定点)  /     \      (集成周稳定点)   /
dev        ●──●──●──●──●──●───────●──●──●──●──●──●──●──●──●──────────►
           │\    /│\    /        │\           /
feature/…  ●──●   ●──●           ●──●──●──●──●   ← 各人个人分支从 dev 切出、合回 dev
```

### 常用命令

```bash
# 从最新 dev 切出个人分支
git checkout dev
git pull origin dev
git checkout -b feature/yuhao-max30102

# 开发中同步 dev 的更新（rebase 保持历史线性）
git fetch origin
git rebase origin/dev
#   冲突：逐个文件解决 → git add <file> → git rebase --continue
#   放弃本次 rebase：git rebase --abort

# 推送个人分支
git push origin feature/yuhao-max30102        # 首次
git push --force-with-lease                    # rebase 之后（安全强推，不覆盖他人提交）

# 合并回 dev
git checkout dev
git pull origin dev
git merge --no-ff feature/yuhao-max30102
git push origin dev
```

> 不熟悉 rebase 也可用 `git merge origin/dev` 同步，效果相同，仅历史多出合并节点，不影响功能。

### 冲突处理

- 冲突标记 `<<<<<<<`、`=======`、`>>>>>>>` 之间是两边版本，手工取舍后**删除全部标记**再 `git add`。
- 不确定的冲突（尤其公共文件）不要臆测，先在群里对齐再改。

### 减少冲突的约定

这是异地并行开发能否顺利集成的关键：

- 各人分支**尽量只改自己负责的目录**：`modules/` 下自己的子目录、`algorithms/` 中自己负责的文件。
- **公共契约改动前必须先同步**：`include/`（接口头文件）与 `docs/引脚分配表.md`（引脚表）的任何改动，
  须先在群里说明并经三人确认，避免多人同时改同一份契约造成难解冲突。

## 2. 每周合并节奏

约定**每周日为合并日**，统一节奏、集中处理集成问题。每人流程：

1. `git fetch origin` 后 `git rebase origin/dev`，将分支追到最新 dev；
2. 本地运行 `ctest --preset host`，**全绿**才继续；
3. 合并到 `dev`。

组长职责：各人合入后确认 `dev` 上 host 测试仍全绿，择稳定点合入 `main`。

合并前检查清单：

- [ ] 本地能完整编译，无报错；
- [ ] `ctest --preset host` 全部通过；
- [ ] 自己模块的**单体实测数据已更新**（数值/日志记入提交或 docs）；
- [ ] 无无关文件混入（build 产物、临时文件、个人 IDE 配置，应由 `.gitignore` 排除）；
- [ ] 未擅改公共契约（如需改动，已完成三人确认）。

## 3. 提交信息规范

采用 Conventional Commits 简化版。清晰的提交信息便于回溯与二分定位问题。格式：

```
<type>(<模块>): <简述>

<可选正文：中文说明动机 / 实测结论>
```

| type | 含义 |
| --- | --- |
| `feat` | 新功能 |
| `fix` | 修 bug |
| `docs` | 文档 |
| `test` | 测试 |
| `refactor` | 重构（不改行为） |
| `chore` | 杂务（配置、脚手架等） |
| `build` | 构建系统 / 依赖 |

示例：

- `feat(max30102): 实现心率峰值检测，单体实测 72bpm`
- `fix(hx711_weight): 修正标定系数符号错误`
- `docs(引脚分配表): 冻结整机引脚分配`
- `build(cmake): 新增 host preset，支持 ctest 单元测试`

要求：**每个提交尽量原子、可独立编译**——一次提交只做一件事，且提交后代码仍能编译，
便于回溯与二分定位。

## 4. 里程碑 tag

在 `main` 的稳定点打附注 tag，对应两个关键节点：

| tag | 对应 | 验收要求 |
| --- | --- | --- |
| `v0.5-midterm` | 8.10 中期检查 | 各模块单体可演示 + 有实测数据；host 测试全绿；整机能空跑。 |
| `v1.0-final` | 9.8 最终验收 | 整机 8 项测量 + 综合评分 + BLE 看板 + 历史存储全部打通。 |

```bash
git checkout main && git pull origin main

git tag -a v0.5-midterm -m "中期检查：各模块单体可演示，含实测数据，host 全绿"
git push origin v0.5-midterm

git tag -a v1.0-final -m "最终验收：8 项测量 + 评分 + 看板 + 历史全打通"
git push origin v1.0-final
```

## 6. 代码评审约定

- **公共契约的改动须至少一名其他成员 review 后方可合入**，包括
  `include/health_if.h`、`include/health_record.h`、`algorithms/` 公共库、`docs/引脚分配表.md`。
- **纯自己模块内部实现**可自审后合入 `dev`，前提是不破坏整体编译（合并前务必本地编译 + host 测试通过）。
- Review 关注点：接口是否与约定一致、是否影响他人模块、是否有实测数据支撑。

---

## 7. 集成周注意事项（9 月）

- 集中集成时切 `feature/integration` 或直接在 `dev` 联调，稳定后再合入 `main`。
- 用 `MODULE_SET=full` 编译整机，做全模块联合验证。
- **接线以 `docs/引脚分配表.md` 为准**：现场如需改线，先改表、群里同步，再动硬件。
- 每天收尾时在 `dev` 打一个可回退的稳定点，避免联调改乱后无法回溯。
