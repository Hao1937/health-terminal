# 健康体测终端 · 接口文档

> 本文档正文为简体中文，代码标识符保持英文，内容与 `include/health_if.h`、`include/health_record.h`、`algorithms/record_codec.c`、`algorithms/crc16.c` 逐字一致。

---

## 1. 概述

本文档是全队**第一周冻结的接口契约**，是各模块并行开发的公共基线。它约定了统一状态码、统一传感器接口、核心数据结构 `measurement_record_t`、UART/BLE 数据帧格式与 CRC16 算法。

任何对本契约的改动都必须满足以下条件，否则不得合入 `dev`：

1. 经**三人评审**通过；
2. 同步更新 `include/` 下的头文件（`health_if.h` / `health_record.h`），并在必要时递增 `RECORD_VERSION` / `FRAME_VERSION`；
3. 同步更新 `web/dashboard.html` 中的解析逻辑（`parseFrame` / `parseRecord`），保证下位机与 Web 看板严格一致。

所有物理量一律采用**定点整数**表示，避免在 `-Os` 优化下引入软浮点，也便于 CRC 计算与跨平台（STM32 / x86 / 安卓）保持一致。

## 2. 统一状态码 hs_status_t

来自 `include/health_if.h`。所有 `init` / `measure` 函数都返回此枚举。

| 枚举值 | 数值 | 语义 |
| --- | --- | --- |
| `HS_OK` | 0 | 成功，数据有效 |
| `HS_NOT_READY` | 1 | 传感器/数据尚未就绪，调用方应稍后重试 |
| `HS_UNSTABLE` | 2 | 读到数据但未达稳定判据（如体重未稳、姿态抖动） |
| `HS_TIMEOUT` | 3 | 等待超时（如 HC-SR04 无回波、HX711 长期不 ready） |
| `HS_NOT_IMPLEMENTED` | 4 | 模块桩：功能尚未实现，仅占位 |

### 调用方处理建议

| 状态码 | 调用方应如何处理 |
| --- | --- |
| `HS_OK` | 采纳 `hs_sample_t` 中的数据，写入对应 record 字段 |
| `HS_NOT_READY` | 不写值，延时后重试；超过重试上限则按超时处理 |
| `HS_UNSTABLE` | 继续保持采集，等待数据稳定；UI 可提示“请保持静止/稳定” |
| `HS_TIMEOUT` | 放弃本次采集，对应指标填 `HS_VALUE_INVALID`，UI 显示 `--` |
| `HS_NOT_IMPLEMENTED` | 视为该项暂不可用，指标填 `HS_VALUE_INVALID`，不阻塞整机流程 |

> 提示：`hs_status_str(hs_status_t s)` 返回状态码可读名字，供调试串口打印使用。

## 3. 统一传感器接口

### 设计原则

- 所有模块共用同一套状态码 `hs_status_t`；
- 每个测量项实现**统一签名**的两个函数：

```c
hs_status_t xxx_init(void);              /* 上电初始化，可为 NULL 表示无需初始化 */
hs_status_t xxx_measure(hs_sample_t *out); /* 采集一次，写入 hs_sample_t */
```

- `measure` 结果写入通用结构 `hs_sample_t`：

```c
typedef struct {
    int32_t primary;    /* 主指标定点值，单位见对应 record 字段注释 */
    int32_t secondary;  /* 次指标（如 SpO2），无则填 HS_VALUE_INVALID */
} hs_sample_t;
```

大多数传感器只用 `primary`；**双输出传感器**（如 MAX30102 心率+血氧）用 `secondary` 存放 SpO2。

### 测量项枚举 hs_item_t

顺序与 `measurement_record_t` 中指标字段顺序一致：

| 枚举 | 含义 | 负责 |
| --- | --- | --- |
| `HS_ITEM_HEIGHT` | 身高 | 查樊听 |
| `HS_ITEM_WEIGHT` | 体重 | 查樊听 |
| `HS_ITEM_BODY` | BMI/体脂率（由身高体重导出） | 算法（王宇浩） |
| `HS_ITEM_HR_SPO2` | 心率+血氧 | 王宇浩 |
| `HS_ITEM_BALANCE` | 平衡晃动指数 | 刘晏铭 |
| `HS_ITEM_GRIP` | 握力 | 王宇浩 |
| `HS_ITEM_REACTION` | 反应时间 | 刘晏铭 |

> `HS_ITEM_COUNT` 等于有效测量项数量（7），仅用于计数。

### 注册表 hs_sensor_t

app 层用一张 `hs_sensor_t` 注册表，把「测量项」映射到「函数指针」，状态机按项遍历采集，不需要为每个传感器写分支：

```c
typedef struct {
    hs_item_t        item;
    const char      *name;    /* 短名，用于 UI 与调试打印 */
    const char      *owner;   /* 负责人，与头文件 @owner 一致 */
    hs_init_fn_t     init;    /* 上电初始化，可为 NULL 表示无需初始化 */
    hs_measure_fn_t  measure; /* 采集一次，写入 hs_sample_t */
} hs_sensor_t;

extern const hs_sensor_t g_hs_registry[];  /* 在 app/app_sensors.c 中定义 */
```

全局注册表 `g_hs_registry` 定义于 `app/app_sensors.c`，数组以 `item == HS_ITEM_COUNT` 或 `measure == NULL` 的哨兵项结尾。

### 桩要求与替换示例

每个新模块**必须先提供返回 `HS_NOT_IMPLEMENTED` 的桩**，占位后再逐步把实现推进到 `HS_OK`。

桩骨架：

```c
/* modules/xxx/xxx.c —— 初始版（桩） */
#include "health_if.h"

hs_status_t xxx_init(void)
{
    return HS_NOT_IMPLEMENTED;   /* 占位 */
}

hs_status_t xxx_measure(hs_sample_t *out)
{
    (void)out;
    return HS_NOT_IMPLEMENTED;   /* 占位 */
}
```

替换成真实现（以某测量项为例）：

```c
hs_status_t xxx_init(void)
{
    /* 初始化外设：I2C/GPIO/定时器等 */
    if (/* 探测失败 */ 0) return HS_TIMEOUT;
    return HS_OK;
}

hs_status_t xxx_measure(hs_sample_t *out)
{
    if (/* 数据未就绪 */ 0)  return HS_NOT_READY;
    if (/* 未达稳定判据 */ 0) return HS_UNSTABLE;

    out->primary   = /* 主指标定点值，如 height_mm */ 0;
    out->secondary = HS_VALUE_INVALID;  /* 无次指标时填无效值 */
    return HS_OK;
}
```

替换实现时只改模块内部，`g_hs_registry` 中的函数指针无需变动，接口签名保持不变。

## 4. 数据结构 measurement_record_t

**第一周必须冻结。** 结构体总大小固定 **52 字节**，所有多字节整数字段一律**小端**。头文件带编译期断言 `sizeof(measurement_record_t) == 52`，防止编译器擅自填充导致跨端不一致。

```c
typedef struct {
    uint16_t version;        /* = RECORD_VERSION */
    uint16_t reserved;       /* 对齐/预留，写入 0 */
    uint32_t timestamp;      /* 上电以来的秒数或 RTC 秒；本项目用 SysTick 秒计数 */

    int32_t  height_mm;
    int32_t  weight_g;
    int32_t  bmi_x100;
    int32_t  bodyfat_x10;
    int32_t  heart_rate_bpm;
    int32_t  spo2_x10;
    int32_t  balance_x10;
    int32_t  grip_kg_x10;
    int32_t  reaction_ms;

    int32_t  score;

    uint16_t crc16;
    uint16_t pad;            /* 对齐到 4 字节边界，写入 0 */
} measurement_record_t;
```

### 完整字节偏移表

| 偏移 | 类型 | 字段名 | 单位与缩放 | 示例 |
| --- | --- | --- | --- | --- |
| 0 | `uint16` | `version` | = `0x01`（`RECORD_VERSION`） | `0x01` |
| 2 | `uint16` | `reserved` | 预留，写入 0 | `0` |
| 4 | `uint32` | `timestamp` | 上电秒数 | — |
| 8 | `int32` | `height_mm` | 身高 mm | `1732` = 173.2cm |
| 12 | `int32` | `weight_g` | 体重 g | `65300` = 65.3kg |
| 16 | `int32` | `bmi_x100` | BMI×100 | `2180` = 21.80 |
| 20 | `int32` | `bodyfat_x10` | 体脂率×10 百分数 | `185` = 18.5% |
| 24 | `int32` | `heart_rate_bpm` | 心率 bpm | — |
| 28 | `int32` | `spo2_x10` | 血氧×10 百分数 | `985` = 98.5% |
| 32 | `int32` | `balance_x10` | 平衡晃动指数×10（越小越稳） | — |
| 36 | `int32` | `grip_kg_x10` | 握力×10 kg | `352` = 35.2kg |
| 40 | `int32` | `reaction_ms` | 反应时间 ms | — |
| 44 | `int32` | `score` | 综合评分 0~100 | — |
| 48 | `uint16` | `crc16` | CRC16-CCITT，覆盖前 48 字节 | — |
| 50 | `uint16` | `pad` | 对齐，写入 0 | `0` |

### 说明

- **无效指标**填 `HS_VALUE_INVALID = 0x7FFFFFFF`，UI 层显示为 `--`。
- 所有物理量用**定点整数**避免软浮点，UI 层负责还原为带小数的显示值。
- record 的 `crc16` 为 **CRC16-CCITT**，覆盖前 **48 字节**（从 `version` 到 `score`，即除 `crc16` 与 `pad` 自身外的全部字节）。对应宏 `RECORD_CRC_LEN = sizeof(measurement_record_t) - 4 = 48`。
- 编码/校验参考 `algorithms/record_codec.c` 的 `record_finalize()`（写入 version/reserved/pad 并计算 crc16）与 `record_verify()`（校验 version 与 crc16）。

## 5. UART/BLE 数据帧格式

**第一周必须冻结。** 帧布局（全部小端）：

```
[SOF0=0xAA][SOF1=0x55][TYPE:1][LEN:2 小端][PAYLOAD:LEN][CRC:2 小端]
```

| 字段 | 长度 | 说明 |
| --- | --- | --- |
| `SOF0` / `SOF1` | 2 | 固定 `0xAA 0x55`，用于接收端**重同步** |
| `TYPE` | 1 | 帧类型，见下表 |
| `LEN` | 2 | PAYLOAD 字节数（小端 `uint16`） |
| `PAYLOAD` | LEN | 按 TYPE 解释 |
| `CRC` | 2 | CRC16-CCITT 覆盖 `[TYPE, LEN, PAYLOAD]`（**不含 SOF**），小端 |

帧类型 `frame_type_t`：

| 类型 | 值 | PAYLOAD |
| --- | --- | --- |
| `FRAME_TYPE_RECORD` | `0x01` | `measurement_record_t`，52 字节 |
| `FRAME_TYPE_ACK` | `0x02` | 0 字节 |
| `FRAME_TYPE_HELLO` | `0x10` | ASCII 版本串，握手/自检用 |

相关常量：`FRAME_HEADER_LEN = 5`（SOF0+SOF1+TYPE+LEN），`FRAME_CRC_LEN = 2`，`FRAME_OVERHEAD = 7`。因此一条 RECORD 帧总长 = 7 + 52 = **59 字节**。

### 一帧完整字节示例（TYPE=0x01, LEN=52）

```
偏移 0    : AA               SOF0
偏移 1    : 55               SOF1
偏移 2    : 01               TYPE = FRAME_TYPE_RECORD
偏移 3-4  : 34 00            LEN = 52（0x0034，小端）
偏移 5-56 : ... 52 字节 ...  PAYLOAD = measurement_record_t 原始字节
偏移 57-58: LL HH            CRC16-CCITT([TYPE,LEN,PAYLOAD])，小端
```

CRC 覆盖范围为从 `TYPE`（偏移 2）起、长度 `1 + 2 + 52 = 55` 字节，与 `record_codec.c` 中 `frame_encode_record()` 的实现一致。

### 字节序说明（重点）

- **STM32F103 与 x86/安卓均为小端（little-endian）。** record 与 frame 的所有多字节整数字段一律以小端在 Flash / BLE / UART 上收发，无需字节交换，可**直接按小端整块拷贝**（见 `frame_encode_record` 中对 record 字节的直接拷贝）。
- **Web 端**用 `DataView` 解析，并且 `littleEndian = true`。
- `web/dashboard.html` 的 `parseFrame` / `parseRecord` **必须与本节严格一致**：SOF 重同步、TYPE 判定、LEN 小端读取、CRC 覆盖范围（不含 SOF）、以及 record 各字段偏移与缩放。

## 6. CRC16 算法

采用 **CRC16-CCITT**：

- 多项式 `poly = 0x1021`
- 初值 `init = 0xFFFF`
- **不反转**输入、**不反转**输出（no reflect in/out），无最终异或

### 已知测试向量

```
"123456789" → 0x29B1
```

任何 C / JS 实现都应通过此向量校验。

### C 版本要点（`algorithms/crc16.c`）

```c
uint16_t crc16_ccitt(const uint8_t *buf, size_t len)
{
    uint16_t crc = 0xFFFFu;
    for (size_t i = 0; i < len; ++i) {
        crc ^= (uint16_t)buf[i] << 8;      /* 高字节对齐 */
        for (int b = 0; b < 8; ++b) {
            if (crc & 0x8000u) {
                crc = (uint16_t)((crc << 1) ^ 0x1021u);
            } else {
                crc = (uint16_t)(crc << 1);
            }
        }
    }
    return crc;
}
```

### JS 版本要点（与 `web/dashboard.html` 一致）

```js
function crc16_ccitt(bytes) {
  let crc = 0xFFFF;
  for (const byte of bytes) {
    crc ^= byte << 8;
    for (let b = 0; b < 8; b++) {
      crc = (crc & 0x8000) ? ((crc << 1) ^ 0x1021) : (crc << 1);
      crc &= 0xFFFF;               // 关键：每步截断到 16 位
    }
  }
  return crc & 0xFFFF;
}
```

> 要点：字节先左移 8 位与 `crc` 异或；逐位处理时按最高位决定是否异或 `0x1021`；JS 因无 16 位整型，每步需 `& 0xFFFF` 截断。两处（record 的 `crc16`、frame 的 `CRC`）复用同一函数，仅覆盖范围不同。
