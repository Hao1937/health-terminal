# 接口文档

> 本文档定义全队共享的软件接口：统一状态码、传感器函数签名、测量记录结构、
> UART/BLE 数据帧格式与 CRC16 算法。内容与 `include/health_if.h`、`include/health_record.h`、
> `algorithms/record_codec.c`、`algorithms/crc16.c` 严格对应。正文中文，代码标识符保持英文。

---

## 1. 为什么先冻结接口

三人异地并行开发，各模块最终要拼成一台整机。若接口不统一，集成时会大面积返工。
因此第一周先把以下五项**冻结**（约定后不轻易改动）：状态码、传感器函数签名、
`measurement_record_t` 结构、数据帧格式、CRC16。这样每个人只要遵守接口，各自实现互不影响。

冻结项确需修改时，必须同时满足：

1. 经其他人评审通过；
2. 同步更新 `include/` 下头文件，改动数据结构时递增 `RECORD_VERSION` / `FRAME_VERSION`；
3. 同步更新 `web/dashboard.html` 的解析逻辑（`parseFrame` / `parseRecord`），保证上下位机一致。

**所有物理量统一用定点整数表示**（如身高存毫米、BMI 存放大 100 倍的整数）。这样做有三个目的：
避免在单片机上引入软浮点、简化 CRC 计算、并让 STM32 与 PC / 安卓三端结果完全一致。

## 2. 统一状态码 hs_status_t

定义于 `include/health_if.h`。所有 `init` / `measure` 函数都返回该枚举，用来告知调用方本次采集的结果。

| 枚举值 | 数值 | 含义 |
| --- | --- | --- |
| `HS_OK` | 0 | 成功，数据有效 |
| `HS_NOT_READY` | 1 | 尚未就绪，应稍后重试 |
| `HS_UNSTABLE` | 2 | 读到数据但未达稳定判据（体重晃动、姿态抖动等） |
| `HS_TIMEOUT` | 3 | 等待超时（HC-SR04 无回波、HX711 长期不 ready 等） |
| `HS_NOT_IMPLEMENTED` | 4 | 桩函数，功能未实现，仅占位 |

状态码的意义在于让调用方（app 主状态机）无需了解传感器细节即可统一决策：

| 状态码 | 调用方处理 |
| --- | --- |
| `HS_OK` | 采纳数据，写入对应 record 字段 |
| `HS_NOT_READY` | 暂不写值，延时后重试；超过重试上限按超时处理 |
| `HS_UNSTABLE` | 继续采集等待稳定；UI 可提示"请保持静止" |
| `HS_TIMEOUT` | 放弃本次，对应字段填 `HS_VALUE_INVALID`，UI 显示 `--` |
| `HS_NOT_IMPLEMENTED` | 视为该项暂不可用，填 `HS_VALUE_INVALID`，不阻塞整机流程 |

> `hs_status_str(hs_status_t s)` 可将状态码转为可读字符串，供调试串口打印。

## 3. 统一传感器接口

设计目标：让所有传感器对上层呈现完全一致的形态，使 app 能用同一套流程遍历采集，
而无需为每个传感器编写专用分支。

每个测量项实现签名一致的两个函数：

```c
hs_status_t xxx_init(void);                /* 上电初始化；无需初始化时直接返回 HS_OK */
hs_status_t xxx_measure(hs_sample_t *out); /* 采集一次，结果写入 out */
```

采集结果统一写入 `hs_sample_t`：

```c
typedef struct {
    int32_t primary;    /* 主指标定点值，单位见对应 record 字段注释 */
    int32_t secondary;  /* 次指标（如血氧）；无则填 HS_VALUE_INVALID */
} hs_sample_t;
```

多数传感器只用 `primary`。`secondary` 专为**单次输出两个值**的传感器保留，例如
MAX30102 同时给出心率与血氧，血氧存于 `secondary`。

### 测量项枚举 hs_item_t

顺序与 `measurement_record_t` 的指标字段顺序一致，便于遍历时对号入座：

| 枚举 | 含义 | 负责人 |
| --- | --- | --- |
| `HS_ITEM_HEIGHT` | 身高 | 查樊听 |
| `HS_ITEM_WEIGHT` | 体重 | 查樊听 |
| `HS_ITEM_BODY` | BMI/体脂率（由身高体重导出） | 王宇浩 |
| `HS_ITEM_HR_SPO2` | 心率 + 血氧 | 王宇浩 |
| `HS_ITEM_BALANCE` | 平衡晃动指数 | 刘晏铭 |
| `HS_ITEM_GRIP` | 握力 | 王宇浩 |
| `HS_ITEM_REACTION` | 反应时间 | 刘晏铭 |

> `HS_ITEM_COUNT` 等于有效测量项数量（7），仅用于计数。

### 注册表 hs_sensor_t

注册表把「测量项」映射到「函数指针」，是"用同一套流程遍历所有传感器"的实现载体：
主状态机只需遍历该表逐项调用，新增/替换传感器都不改动状态机代码。

```c
typedef struct {
    hs_item_t        item;
    const char      *name;    /* 短名，用于 UI 与调试打印 */
    const char      *owner;   /* 负责人 */
    hs_init_fn_t     init;    /* 初始化；无需初始化填 NULL */
    hs_measure_fn_t  measure; /* 采集一次 */
} hs_sensor_t;

extern const hs_sensor_t g_hs_registry[];  /* 定义于 app/app_sensors.c */
```

`g_hs_registry` 以 `item == HS_ITEM_COUNT` 或 `measure == NULL` 的哨兵项结尾，作为遍历终止标记。

### 桩优先原则

每个模块**先提交返回 `HS_NOT_IMPLEMENTED` 的桩**，使整机在功能未完成时也能编译联调，
再将实现逐步推进到 `HS_OK`。这是"整机始终可运行"的关键约定。

桩骨架：

```c
/* modules/xxx/xxx.c —— 初始版（桩） */
#include "health_if.h"

hs_status_t xxx_init(void)    { return HS_NOT_IMPLEMENTED; }
hs_status_t xxx_measure(hs_sample_t *out) { (void)out; return HS_NOT_IMPLEMENTED; }
```

替换为真实现（骨架）：

```c
hs_status_t xxx_init(void)
{
    /* 初始化外设：I2C / GPIO / 定时器等 */
    if (/* 探测失败 */ 0) return HS_TIMEOUT;
    return HS_OK;
}

hs_status_t xxx_measure(hs_sample_t *out)
{
    if (/* 数据未就绪 */ 0)  return HS_NOT_READY;
    if (/* 未达稳定判据 */ 0) return HS_UNSTABLE;

    out->primary   = /* 主指标定点值，如 height_mm */ 0;
    out->secondary = HS_VALUE_INVALID;  /* 无次指标时填无效值 */
    return HS_OK;
}
```

替换实现只改模块内部，注册表中的函数指针与接口签名均不变，因此不影响其他模块。

## 4. 测量记录 measurement_record_t（冻结项）

一次完整测量的载体，也是 Flash 存储与 BLE 传输的基本单元，**固定 52 字节**、多字节字段一律**小端**。
头文件含编译期断言 `sizeof(measurement_record_t) == 52`，防止编译器因对齐填充改变结构大小、
导致跨端解析错位。

```c
typedef struct {
    uint16_t version;        /* = RECORD_VERSION */
    uint16_t reserved;       /* 对齐预留，写 0 */
    uint32_t timestamp;      /* 上电以来秒数（本项目取 SysTick 秒计数） */

    int32_t  height_mm;
    int32_t  weight_g;
    int32_t  bmi_x100;
    int32_t  bodyfat_x10;
    int32_t  heart_rate_bpm;
    int32_t  spo2_x10;
    int32_t  balance_x10;
    int32_t  grip_kg_x10;
    int32_t  reaction_ms;

    int32_t  score;

    uint16_t crc16;
    uint16_t pad;            /* 对齐到 4 字节，写 0 */
} measurement_record_t;
```

字段偏移与缩放（存 Flash、发 BLE 均以此为准）：

| 偏移 | 类型 | 字段名 | 单位与缩放 | 示例 |
| --- | --- | --- | --- | --- |
| 0 | `uint16` | `version` | = `0x01`（`RECORD_VERSION`） | `0x01` |
| 2 | `uint16` | `reserved` | 预留，写 0 | `0` |
| 4 | `uint32` | `timestamp` | 上电秒数 | — |
| 8 | `int32` | `height_mm` | 身高 mm | `1732` = 173.2cm |
| 12 | `int32` | `weight_g` | 体重 g | `65300` = 65.3kg |
| 16 | `int32` | `bmi_x100` | BMI×100 | `2180` = 21.80 |
| 20 | `int32` | `bodyfat_x10` | 体脂率×10（百分数） | `185` = 18.5% |
| 24 | `int32` | `heart_rate_bpm` | 心率 bpm | — |
| 28 | `int32` | `spo2_x10` | 血氧×10（百分数） | `985` = 98.5% |
| 32 | `int32` | `balance_x10` | 晃动指数×10（越小越稳） | — |
| 36 | `int32` | `grip_kg_x10` | 握力×10 kg | `352` = 35.2kg |
| 40 | `int32` | `reaction_ms` | 反应时间 ms | — |
| 44 | `int32` | `score` | 综合评分 0~100 | — |
| 48 | `uint16` | `crc16` | CRC16-CCITT，覆盖前 48 字节 | — |
| 50 | `uint16` | `pad` | 对齐，写 0 | `0` |

说明：

- 无效指标填 `HS_VALUE_INVALID = 0x7FFFFFFF`，UI 层显示为 `--`。
- 各字段为定点整数，UI 层负责还原为带小数的显示值（如 `bmi_x100 / 100`）。
- `crc16` 采用 CRC16-CCITT，覆盖前 **48 字节**（`version` 至 `score`，不含 `crc16`、`pad` 自身），
  对应宏 `RECORD_CRC_LEN = 48`。
- 打包与校验统一调用 `algorithms/record_codec.c` 的 `record_finalize()`（填写 version/pad 并计算
  crc16）与 `record_verify()`（校验 version 与 crc16），不要各自手写，以免实现不一致。

## 5. UART/BLE 数据帧格式（冻结项）

record 通过串口/蓝牙传输时需要封装成帧，以便接收端定位边界、判断类型、校验完整性。帧布局（均小端）：

```
[SOF0=0xAA][SOF1=0x55][TYPE:1][LEN:2 小端][PAYLOAD:LEN][CRC:2 小端]
```

| 字段 | 长度 | 作用 |
| --- | --- | --- |
| `SOF0` / `SOF1` | 2 | 固定 `0xAA 0x55`，供接收端重同步定位帧头 |
| `TYPE` | 1 | 帧类型，决定 PAYLOAD 的解释方式 |
| `LEN` | 2 | PAYLOAD 字节数（小端 `uint16`） |
| `PAYLOAD` | LEN | 帧数据，按 TYPE 解释 |
| `CRC` | 2 | CRC16-CCITT，覆盖 `[TYPE, LEN, PAYLOAD]`（**不含 SOF**），小端 |

帧类型 `frame_type_t`：

| 类型 | 值 | PAYLOAD |
| --- | --- | --- |
| `FRAME_TYPE_RECORD` | `0x01` | `measurement_record_t`，52 字节 |
| `FRAME_TYPE_ACK` | `0x02` | 0 字节 |
| `FRAME_TYPE_HELLO` | `0x10` | ASCII 版本串，握手/自检用 |

相关常量：`FRAME_HEADER_LEN = 5`、`FRAME_CRC_LEN = 2`、`FRAME_OVERHEAD = 7`。
故一条 RECORD 帧总长 = 7 + 52 = **59 字节**。

RECORD 帧字节示例（TYPE=0x01, LEN=52）：

```
偏移 0    : AA               SOF0
偏移 1    : 55               SOF1
偏移 2    : 01               TYPE = FRAME_TYPE_RECORD
偏移 3-4  : 34 00            LEN = 52（0x0034，小端）
偏移 5-56 : ... 52 字节 ...  PAYLOAD = record 原始字节
偏移 57-58: LL HH            CRC16-CCITT，小端
```

CRC 从 `TYPE`（偏移 2）起算，长度 `1 + 2 + 52 = 55` 字节，与 `frame_encode_record()` 一致。

### 字节序

STM32F103、x86、安卓均为小端，因此 record 与帧中的整数可直接整块拷贝收发、无需字节交换。
Web 端用 `DataView` 解析时须设 `littleEndian = true`。`web/dashboard.html` 的
`parseFrame` / `parseRecord` 必须与本节完全一致（SOF 重同步、TYPE 判定、LEN 小端、CRC 覆盖范围、
各字段偏移与缩放）。

## 6. CRC16 算法

用于 record 与帧的完整性校验，采用 **CRC16-CCITT**。参数如下（C 与 JS 两端须一致）：

- 多项式 `poly = 0x1021`
- 初值 `init = 0xFFFF`
- 输入输出均不反转（no reflect in/out），无最终异或

标准测试向量：`"123456789"` → `0x29B1`。两端实现均应通过此向量后再使用。

C 版（`algorithms/crc16.c`）：

```c
uint16_t crc16_ccitt(const uint8_t *buf, size_t len)
{
    uint16_t crc = 0xFFFFu;
    for (size_t i = 0; i < len; ++i) {
        crc ^= (uint16_t)buf[i] << 8;      /* 高字节对齐 */
        for (int b = 0; b < 8; ++b)
            crc = (crc & 0x8000u) ? (uint16_t)((crc << 1) ^ 0x1021u)
                                  : (uint16_t)(crc << 1);
    }
    return crc;
}
```

JS 版（与 `web/dashboard.html` 一致）：

```js
function crc16_ccitt(bytes) {
  let crc = 0xFFFF;
  for (const byte of bytes) {
    crc ^= byte << 8;
    for (let b = 0; b < 8; b++) {
      crc = (crc & 0x8000) ? ((crc << 1) ^ 0x1021) : (crc << 1);
      crc &= 0xFFFF;               // JS 无 16 位整型，每步须截断到 16 位
    }
  }
  return crc & 0xFFFF;
}
```

record 的 `crc16` 与帧的 `CRC` 复用同一函数，仅覆盖范围不同。
